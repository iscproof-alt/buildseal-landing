How it works

1. Seal your release uses: hakannbjk55-afk/buildseal-action@v1 Add to your CI workflow. BuildSeal produces an evidence pack on every release.
2. Share the verification link verify.buildseal.io/release/<seal_id> Anyone can open the link and see the verification report.
3. Verify independently ./isc_verify evidence_pack.tar Download the pack and binary. Verify offline — no BuildSeal required.

Design Principles

1. Verification is always offline — no BuildSeal, no internet required.
2. Pack format is open — anyone can implement a verifier.
3. Production is paid, verification is free — forever.

FAQ

What does BuildSeal do?

Signs your CI release and produces a portable evidence pack. Answers "was this release clean?" months later.

Why not logs?

Logs disappear when services shut down. A pack stays with you — no internet, no account, no BuildSeal needed.

Why not Sigstore?

Sigstore requires the Rekor log to verify. BuildSeal packs work offline. No external dependency.

What does offline verify mean?

Download the binary and the pack. Run locally. No internet required, no need to trust BuildSeal.

What does it prove? What doesn't it prove?

Proves: artifact unchanged, authorized key signed, derived from this commit. Does not prove: build environment was not manipulated.

How does key management work?

Ed25519 key stored in CI secrets. Governance — including rotation and revocation — is embedded in the pack itself.

What if BuildSeal shuts down?

Your pack and the isc_verify binary stay with you. You can verify forever.

Is this enough for SOC2 audit?

BuildSeal provides evidence. The auditor downloads the pack and verifies independently. The audit decision is yours.

Does it work outside GitHub Actions?

GitHub Action is available now. Jenkins, GitLab, and Azure DevOps are on the roadmap.

Is it free?

Verification is free forever. Pack production via BuildSeal API — pricing coming soon.
